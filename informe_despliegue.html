Proyecto: Sistema Web "D'licias Fast Food" (Broastería)

El presente documento detalla la topología del sistema, el proceso manual de migración inicial, la automatización del flujo de integración continua (CI/CD) y la consolidación de los contenedores Docker en el servidor Azure.

1. Arquitectura General del Sistema

El sistema sigue un modelo de cliente-servidor desacoplado, donde el Frontend (Angular) se aloja en Vercel y el Backend (Spring Boot + MySQL) reside en una Máquina Virtual de Azure, protegido por un proxy inverso.

Justificación del Uso de Docker

Al empaquetar el servidor en contenedores, se evita el problema de compatibilidad de entornos. El Proxy Caddy auto-renueva el SSL para encriptar los datos. El Backend procesa toda la lógica aislando el sistema de archivos, y el Contenedor MySQL se comunica únicamente por la red virtual de Docker interna, impidiendo hackeos externos al puerto 3306.

2. Proceso de Migración Inicial (Primer Despliegue a Azure)

Antes de automatizar, se realizó la migración fundacional del código y la base de datos hacia el servidor virgen de Azure mediante protocolos de transferencia segura (SCP/SSH).

Paso 2.1: Empaquetado y Transferencia de Archivos

Se comprimió el código local y se transfirió utilizando la llave criptográfica asimétrica (.pem):

# 1. Comprimir el backend localmente
tar -czf backend.tar.gz broasteriabackend-main/

# 2. Transferir los archivos críticos al servidor Azure mediante SCP
scp -i produccion_key.pem backend.tar.gz azureuser@172.184.131.149:~/
scp -i produccion_key.pem docker-compose.yml azureuser@172.184.131.149:~/
scp -i produccion_key.pem broasteria.sql azureuser@172.184.131.149:~/

Paso 2.2: Desempaquetado y Primera Ejecución

Dentro del servidor, se descomprimió el código y se ejecutó Docker Compose para construir la imagen e inicializar la base de datos:

# 3. Conectar al servidor mediante SSH
ssh -i produccion_key.pem azureuser@172.184.131.149

# 4. Descomprimir el código fuente
tar -xzf backend.tar.gz
mv broasteriabackend-main broasteriabackend

# 5. Crear contenedor oculto para variables de entorno
nano ~/broasteriabackend/.env

# 6. Construir e inicializar los 3 contenedores (Caddy, Backend, DB)
sudo docker compose up -d --build

3. Arquitectura de Despliegue Continuo (CI/CD)

Para agilizar el desarrollo, se configuró GitHub Actions. Ahora, cada vez que un desarrollador aprueba un cambio local y lo sube al repositorio en la rama main, GitHub se encarga de todo el proceso de migración de forma invisible y automática.

4. Gestión de Secretos y Variables de Entorno

El archivo .env inyectado en el servidor contiene únicamente las 4 variables críticas para el funcionamiento.

4.1. Cómo conectarse al servidor y editar el archivo .env

Para modificar las credenciales del sistema en cualquier momento, el administrador debe conectarse de forma segura al servidor Azure y utilizar un editor de texto en consola (Nano).

sequenceDiagram actor Admin participant Terminal as Consola Local (Windows) participant Azure as Servidor Azure participant Nano as Editor Nano (.env) participant Docker as Contenedor Java Admin->>Terminal: Abre CMD / PowerShell Terminal->>Azure: ssh -i produccion_key.pem azureuser@172.184.131.149 Azure-->>Terminal: Autenticado Terminal->>Azure: nano ~/broasteriabackend/.env Azure->>Nano: Abre editor de consola Admin->>Nano: Cambia variable y guarda (Ctrl+O, Ctrl+X) Terminal->>Azure: sudo docker compose restart backend Azure->>Docker: Reinicia sistema Docker-->>Admin: Nuevas variables aplicadas

Figura 3: Flujo de conexión y edición de variables en producción.

Comandos Exactos a Ejecutar:

# 1. Iniciar sesión en el servidor Azure (la llave .pem debe estar en la carpeta actual)
ssh -i produccion_key.pem azureuser@172.184.131.149

# 2. Abrir el archivo de variables oculto
nano ~/broasteriabackend/.env

# 3. Guardar y salir del editor Nano:
#    a. Presionar: Ctrl + O (Guarda)
#    b. Presionar: Enter (Confirma el nombre del archivo)
#    c. Presionar: Ctrl + X (Sale del editor)

# 4. Reiniciar el contenedor Backend para que absorba los cambios
cd ~/broasteriabackend
sudo docker compose restart backend

# 5. Salir del servidor Azure
exit

4.2. Variables Críticas Actuales

A continuación, las variables configuradas en el entorno:

Variable de Entorno Valor en Producción / Descripción
SQL_INIT_MODE never (Evita que el menú se duplique al reiniciar el contenedor en producción)
GEMINI_API_KEY Llave privada de Groq (Llama-3.1-8b-instant) para procesar las respuestas del chatbot.
SPRING_MAIL_USERNAME Correo origen de D'licias Fast Food (Usado para envío de notificaciones y recuperación)
SPRING_MAIL_PASSWORD Contraseña de aplicación de Gmail para permitir el envío automatizado de correos.

5. Resolución de Incidencias Críticas (IA y BD)

Durante la estabilización, PolloBot experimentó rechazos de cuota (Error 413 y 429). Se identificó que spring.sql.init.mode=always forzaba la inyección de 20 productos de prueba tras cada compilación del contenedor.

Impacto: 880 productos clonados. El prompt enviado a la IA superó los 12,628 tokens de texto.

Comandos de Solución (Limpieza SQL):

# Acceso a la base de datos dentro del contenedor en producción
sudo docker exec broasteria_db mysql -uroot -proot defaultdb -e \
"DELETE p1 FROM producto p1 INNER JOIN producto p2 
 WHERE p1.ID_Producto > p2.ID_Producto AND p1.Nombre = p2.Nombre;"

Para complementar, se parametrizó la API de IA imponiendo requestBody.put("max_tokens", 300); y se suprimió la directiva de inicialización automática en producción (SQL_INIT_MODE=never).
